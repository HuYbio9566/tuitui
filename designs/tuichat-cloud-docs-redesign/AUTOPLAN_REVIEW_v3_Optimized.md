# 360 云文档 · 知识工作台优化方案 v4 自动评审报告

评审对象：`Cloud Docs Redesign v3 Optimized.html`  
评审方法：`360-pmc-autoplan`，串行执行价值澄清 → CEO 战略评审 → 设计评审 → 体验评审 → 交叉验证  
评审模式：自动决策，只有方向性冲突才暂停。本次无必须暂停项。

## Phase 1 · 价值锚点

### 5W1H

| 维度 | 提取结果 |
|---|---|
| Why | 企业文档、知识库、云盘资料分散，员工找答案慢，AI 回答缺少权限、来源和企业上下文。 |
| Who | 普通员工、文档创建者、知识库负责人、协作者、安全/权限管理员。 |
| When | 写文档、查规则、做汇报、归档资料、审阅风险、继续历史工作时。 |
| Where | 桌面端云文档工作台；入口包括首页、我的文档、编辑器、知识库、云盘、知识工具和 AI 助手抽屉。 |
| What | 面向企业私域知识的文档与 AI 问答一体化工作台。 |
| How | 选择知识范围 → 提问/生成/审阅 → 查看引用和过滤记录 → 保存回文档或沉淀到知识库。 |

### 价值锚点声明

```
产品名称：360 云文档 · 企业知识工作台
为谁：需要基于企业私域资料写作、检索、归档和审阅的企业员工与知识负责人
解决什么问题：企业知识分散、难检索、难复用，AI 回答缺少权限边界和可信来源
提供什么价值：把文档、知识库、云盘文件和 AI 问答连接成一个可追溯、可沉淀、权限安全的工作台
成功标准：首次有效提问率 ≥ 60%；待关联文档处理率 ≥ 40%；AI 答案引用采纳率 ≥ 35%；保密过滤误放率趋近 0；TTHW ≤ 2 分钟
差异化定位：不是通用 AI 聊天，而是带企业权限、引用来源和知识沉淀闭环的办公知识入口
```

### 价值 → 评审映射

| 价值要求 | CEO 含义 | 设计含义 | 体验含义 |
|---|---|---|---|
| 私域知识可问答 | ROI 来自找资料、写汇报、查规则的效率提升 | 首页 AI、知识库问答、编辑器侧栏必须形成主链路 | 用户 5 分钟内要完成一次有引用的有效回答 |
| 权限安全与可信来源 | 企业采用门槛取决于权限、审计、误放风险 | 权限、保密、引用不应只是标签，要出现在结果与关键操作中 | 用户必须知道答案来自哪里、过滤了什么、能否分享 |
| 知识沉淀闭环 | 价值不只在问答，还在文档入库和知识库维护 | 待关联、入库、保存到文档需要成为可执行任务 | 问完以后能保存、引用、归档，形成复用资产 |
| 工作台一体化 | 避免做成多个割裂产品 | 首页、我的文档、云盘、知识库、AI 抽屉要共享上下文 | 用户不需要理解系统全貌，也能从当前页面获得价值 |

门控结论：通过。5W1H 完整，价值主张可一句话表达，不是单纯功能堆叠。

价值锚定：CEO 阶段以“企业私域知识可问答 + 可信来源 + 知识沉淀闭环”为判断标准。

| 维度 | 得分 | 关键发现 |
|---|---:|---|
| 战略对齐性 | 8/10 | 方向从“云文档编辑器”升级为“企业知识入口”，战略上成立。 |
| ROI 合理性 | 7/10 | 找资料、写汇报、规则问答有明确效率收益，但缺少量化基线和转化漏斗。 |
| 风险与假设 | 5/10 | 最大风险是权限/引用能力若不真实，可信 AI 定位会塌。 |
| 增长与落地路径 | 6/10 | 首页、文档、知识库都有入口，但先打哪个高频场景还不够聚焦。 |

CEO 总分：26/40

### CEO 关键追问

1. 用户每天打开它的第一理由是什么：问规则、写汇报、还是维护知识库？
2. “有引用、有权限”的答案是否真能做到，还是只是 UI 标签？
3. 待关联文档、知识库维护、AI 问答三件事里，哪个是 MVP 的增长抓手？
4. 如果与飞书文档、腾讯文档、企业微信知识库相比，360 的不可替代点是否足够尖？

### CEO 修复建议

| 优先级 | 修复项 | 改法 |
|---|---|---|
| P0 | 明确 MVP 场景 | 先聚焦“基于企业知识库生成带引用的汇报/规则答案”，其他工具降级为辅助入口。 |
| P0 | 定义可信答案能力边界 | 增加方案级说明：答案必须返回引用来源、权限状态、过滤记录和可分享状态。 |
| P1 | 建立 ROI 指标 | 补充“从提问到可用答案耗时”“答案保存到文档率”“待关联处理率”。 |
| P1 | 降低功能扩散 | 高频入口、知识工具、云盘入库保留，但不要与首页 AI 主链路争第一视觉权重。 |

## Phase 3 · 设计评审

价值锚定：设计评分只看界面是否帮助用户完成“可信问答 → 引用确认 → 保存/沉淀”的核心价值链。

| 维度 | 得分 | 价值锚定检查 | 关键发现 |
|---|---:|---|---|
| 信息层级与视觉流 | 7/10 | 服务“私域知识可问答” | 首页 AI 是主视觉，方向正确；但顶部指标仍像数据摆件，任务优先级不够强。 |
| 交互逻辑与状态管理 | 6/10 | 部分服务“可信来源” | AI 抽屉已有待发送、回答草稿、引用和过滤记录；但主 AI 页面仍缺真实生成结果态、失败态、重试态。 |
| 场景适配与边界 | 7/10 | 服务“工作台一体化” | 覆盖首页、我的文档、编辑器、知识库、云盘；管理员、知识负责人路径仍偏弱。 |
| 可访问性与包容性 | 5/10 | 未充分服务“所有目标用户可获得价值” | 有 aria 和 Esc，但弹窗焦点锁、键盘选择知识库、表格移动端体验不足。 |
| 品牌一致性 | 7/10 | 服务“企业可信办公” | 蓝白 B 端风格统一；Mac 装饰和部分通用 AI 表达削弱 360 企业安全感。 |
| 技术可行性 | 8/10 | 服务“可落地闭环” | 基于单页状态和上下文抽屉可实现；风险在后端权限、引用、过滤接口。 |

设计总分：40/60

### 设计修复建议

| 优先级 | 修复项 | 改法 |
|---|---|---|
| P0 | 主 AI 页面补结果链路 | 在 `renderAi` 增加回答完成态：答案、引用、过滤记录、保存到文档、复制引用、重新生成。 |
| P0 | 顶部指标转任务入口 | “待关联知识库 8”应展示为“8 篇会影响 AI 回答质量”，点击进入待关联视图。 |
| P1 | 我的文档列表增加批处理 | 在“待关联知识库”tab 增加批量关联、AI 推荐归属、忽略此项。 |
| P1 | 权限表达后移到结果 | 首页不放说明卡，权限/保密/引用只在答案和分享动作处呈现。当前方向正确，应继续强化。 |
| P2 | 品牌细节收敛 | 弱化 Mac 窗口装饰，增加 360 企业安全产品的可信提示和审计感。 |

## Phase 4 · 体验评审

用户体验价值链：

| 用户痛点 | 产品价值 | 体验要求 | 关键触点 |
|---|---|---|---|
| 找不到分散资料 | 直接基于私域知识提问 | 首屏一眼能问，默认范围合理 | 首页 AI、知识库选择、常用问法 |
| 不信 AI 答案 | 引用可追溯、权限可解释 | 答案必须展示来源和过滤记录 | AI 抽屉结果、知识库问答、复制引用 |
| 文档无法沉淀 | 问答结果能保存回文档/知识库 | 保存、关联、入库路径短 | 编辑器侧栏、待关联列表、云盘入库 |
| 企业安全约束复杂 | 权限和保密自然融入流程 | 安全提示不打断主链路，但关键动作可确认 | 保密标签、发布检查、过滤记录 |

| Pass | 得分 | 价值锚定检查 | 关键发现 |
|---|---:|---|---|
| 首次接触体验 | 7/10 | 支撑“直接基于私域知识提问” | 用户能在首页直接问，但新手仍不清楚“全部知识库”覆盖哪些资料。 |
| 核心使用旅程 | 7/10 | 支撑“问答 → 引用 → 沉淀” | 首页到 AI 抽屉路径短；主 AI 页面与抽屉体验重复，需统一。 |
| 摩擦点追踪 | 6/10 | 部分支撑“短路径获得答案” | 入口多、工具多，认知负荷偏高；一些动作只有 toast，没有实际下一步。 |
| 惊喜时刻设计 | 6/10 | 部分支撑“差异化可信 AI” | “过滤无权限资料 + 保存到文档”有潜力，但目前只在抽屉样例里出现。 |
| 竞品对标 | 6/10 | 支撑“企业知识入口差异化” | 相比飞书/腾讯文档，知识问答和云盘入库更集中；协作编辑细节不足。 |
| 多角色适配 | 5/10 | 未充分支撑“知识负责人/管理员价值” | 普通员工路径清楚，知识负责人和安全管理员缺少专属任务看板。 |
| 长期使用体验 | 6/10 | 部分支撑“持续沉淀” | 最近文档、快捷访问有回访基础；缺少批量处理、自动提醒、知识健康度。 |
| 企业场景特殊考量 | 7/10 | 支撑“权限安全与可信来源” | 保密过滤、权限内检索、发布检查都有露出；审计和多组织边界不够完整。 |

UX 总分：50/80  
TTHW：约 90-120 秒。Hello World 定义为“用户在首页选择知识库并得到一条带引用/过滤记录的可保存回答”。

### UX 修复建议

| 优先级 | 修复项 | 改法 |
|---|---|---|
| P0 | 统一 AI 体验 | 首页 AI、AI 页面、抽屉不要三套交互。建议统一为“当前页面上下文 + 抽屉结果”，AI 页面作为完整历史/深度问答页。 |
| P0 | 让答案可信成为真实体验 | 生成结果必须可展开引用、查看过滤记录、保存到文档。不要停留在标签或 toast。 |
| P1 | 降低首页认知负荷 | 高频入口保留 4 个以内；其余进入“知识工具”。 |
| P1 | 增加知识负责人路径 | 待关联、待复核、过期内容、热门问题缺口，形成一个可执行维护列表。 |
| P2 | 企业集成体验 | 补 SSO、审批流、通知、审计日志入口的体验说明。 |

## Phase 5 · 交叉验证

| 检查项 | 结论 | 矛盾/风险 |
|---|---|---|
| CEO 要求的可信 AI 是否在设计中体现 | 部分体现 | 抽屉体现了引用和过滤记录，但主 AI 页面仍停留在输入态和标签态。 |
| CEO 的 ROI 假设是否被体验支撑 | 部分支撑 | 首页提问路径短，但保存/沉淀/批量处理不足，ROI 很难从一次问答扩展到持续效率。 |
| 设计方案是否与 UX 的 TTHW 一致 | 基本一致 | 首页可在 2 分钟内触发提问；但若需要真实引用检查，当前交互还缺结果页细节。 |
| 多角色体验是否匹配战略 | 不完全匹配 | 战略上覆盖知识负责人和管理员，但设计/体验主要偏普通员工。 |
| 所有阶段是否服务同一价值锚点 | 是 | 都围绕“私域知识可问答、可信来源、知识沉淀闭环”。 |

### 交叉验证结论

当前方案的主方向成立，但存在一个核心断点：**产品价值说的是“可信企业知识工作台”，但界面最完整的部分仍是“提问入口”，不是“可信答案和沉淀闭环”。**  
下一轮优化不应继续加入口，而应补齐结果、引用、过滤、保存、入库和批处理。

## 最终价值回查

- 修复后的方案是否更清晰服务价值锚点：部分是。首页 AI 和上下文抽屉服务价值，但主 AI 结果闭环仍需补齐。
- 差异化定位是否在 CEO/设计/体验层面体现：部分体现。可信来源和权限过滤已经进入设计语言，但还需要真实结果交互证明。
- 成功指标是否有方案层面的支撑：不足。方案有 TTHW 的基础，但缺少埋点、漏斗和知识沉淀指标。

最终结论：**继续推进，但不要再扩入口。下一轮只补“可信答案结果闭环”和“知识沉淀闭环”。**
